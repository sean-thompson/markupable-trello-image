# Codex Review (Iterative)

Send work to OpenAI Codex for independent review. Supports two modes: **Plan Review** (before implementation) and **Code Review** (after implementation). Claude revises based on Codex feedback and re-submits until Codex approves or 5 rounds elapse.

---

## Selecting the Mode

- If the user passes an argument like `/codex-review plan` or `/codex-review code`, use that mode.
- If there is an implementation plan in the current conversation context (from plan mode or discussion), default to **Plan Review**.
- If there are uncommitted code changes (`git diff` shows output), default to **Code Review**.
- If both apply, ask the user which mode to run.

## Model Selection

Default model: `o3`. If the user specifies a model (e.g. `/codex-review o4-mini`), use that instead. Pass it via `-m <model>`.

---

## Project Context Block

Include this context in **every** prompt sent to Codex so it understands the codebase:

```
This is "Markupable", a Trello Power-Up for annotating image attachments with freehand
drawings, dots, and threaded text notes.

Key constraints:
- Stack: React 17.0.1, TypeScript 4.2, Webpack 5, HTML5 Canvas
- Data stored in Trello pluginData under key 'mkp' with a hard 4096-character JSON limit
- Coordinates normalized to 0-1000 scale, paths simplified with Ramer-Douglas-Peucker
- User IDs mapped to short keys (a, b, c...) for space efficiency
- Image fetching requires OAuth via api.trello.com (not trello.com) to avoid CORS blocks
- Power-Up initializes via window.TrelloPowerUp.initialize() in capabilities.ts
- React components use useProvidedTrello() hook from @optro/ui-react
- All HTML pages generated by HtmlWebpackPlugin from Handlebars templates in templates/
- No test framework currently configured
```

## Mode 1: Plan Review

Use when Claude has produced an implementation plan and wants an independent check.

### Step 1: Generate Temp File Path

```bash
REVIEW_ID=$(printf '%04x%04x' $RANDOM $RANDOM)
PLAN_FILE="/tmp/codex-plan-${REVIEW_ID}.md"
VERDICT_FILE="/tmp/codex-verdict-${REVIEW_ID}.md"
```

### Step 2: Write the Plan

Write the full current plan to `$PLAN_FILE`. If there is no plan in the conversation context, ask the user what they want reviewed.

### Step 3: Send to Codex (Round 1)

```bash
codex exec \
  -m o3 \
  -s read-only \
  -o "$VERDICT_FILE" \
  "Review the implementation plan in ${PLAN_FILE}.

[INSERT PROJECT CONTEXT BLOCK HERE]

Focus on:
1. Correctness - Will this plan achieve the stated goals?
2. Trello API usage - Are pluginData limits, OAuth patterns, and Power-Up APIs used correctly?
3. Risks - Edge cases? Data loss? Race conditions?
4. Missing steps - Anything forgotten?
5. Security - XSS, injection, or credential exposure concerns?

Be specific and actionable. If the plan is solid, end with exactly: VERDICT: APPROVED
If changes are needed, end with exactly: VERDICT: REVISE"
```

### Step 4: Read Verdict

1. Read `$VERDICT_FILE`.
2. Present Codex's feedback to the user:

```
## Codex Review — Round N (model: o3)

[Codex's feedback]
```

3. Check the verdict:
   - **VERDICT: APPROVED** → go to Step 7 (Done)
   - **VERDICT: REVISE** → go to Step 5 (Revise)
   - No clear verdict but feedback is positive → treat as approved
   - Max rounds (5) reached → go to Step 7 with a note

### Step 5: Revise the Plan

Based on Codex's feedback:

1. **Revise the plan** — address each issue Codex raised. Update the plan in the conversation context and rewrite `$PLAN_FILE`.
2. Summarize changes for the user:

```
### Revisions (Round N)
- [What changed and why, one bullet per issue addressed]
```

3. If a revision contradicts the user's explicit requirements, skip it and note why.

### Step 6: Re-submit to Codex (Rounds 2–5)

Use a fresh `codex exec` call each round (more reliable than session resume):

```bash
codex exec \
  -m o3 \
  -s read-only \
  -o "$VERDICT_FILE" \
  "This is round N of an iterative plan review. I revised the plan based on your prior feedback.

Changes made:
[List specific changes]

The updated plan is in ${PLAN_FILE}. Please re-review with the same criteria.

If the plan is now solid, end with: VERDICT: APPROVED
If more changes needed, end with: VERDICT: REVISE"
```

Then go back to **Step 4**.

### Step 7: Present Final Result

**If approved:**
```
## Codex Review — Final (model: o3)

**Status:** Approved after N round(s)

[Final feedback]

---
The plan has been reviewed and approved by Codex. Ready for your approval to implement.
```

**If max rounds reached:**
```
## Codex Review — Final (model: o3)

**Status:** Max rounds (5) reached — not fully approved

**Remaining concerns:**
[Unresolved issues from last review]

---
Codex still has concerns. Review the remaining items and decide whether to proceed.
```

### Step 8: Cleanup

```bash
rm -f "$PLAN_FILE" "$VERDICT_FILE"
```

## Mode 2: Code Review

Use after implementation when there are uncommitted changes to review.

### Step 1: Generate Temp File Path

```bash
REVIEW_ID=$(printf '%04x%04x' $RANDOM $RANDOM)
VERDICT_FILE="/tmp/codex-code-review-${REVIEW_ID}.md"
```

### Step 2: Send to Codex (Round 1)

```bash
codex review --uncommitted \
  -m o3 \
  -o "$VERDICT_FILE" \
  "Review these uncommitted changes for a Trello Power-Up called Markupable.

[INSERT PROJECT CONTEXT BLOCK HERE]

Focus on:
1. Correctness - Do the changes work as intended?
2. Data model - Will serialized data fit within the 4096-char pluginData limit?
3. React 17 compatibility - No React 18+ APIs used?
4. Security - XSS, injection, credential exposure?
5. CORS/OAuth - Image fetches go through api.trello.com with proper auth?

If the code is solid, end with: VERDICT: APPROVED
If changes are needed, end with: VERDICT: REVISE"
```

### Step 3: Read Verdict

1. Read `$VERDICT_FILE`.
2. Present Codex's feedback to the user:

```
## Codex Code Review — Round N (model: o3)

[Codex's feedback]
```

3. Check the verdict:
   - **VERDICT: APPROVED** → go to Step 6 (Done)
   - **VERDICT: REVISE** → go to Step 4 (Fix & Re-review)
   - Max rounds (5) → go to Step 6 with a note

### Step 4: Fix the Code

1. Address each issue Codex raised by editing the relevant files.
2. Summarize fixes:

```
### Fixes Applied (Round N)
- [What was fixed and why]
```

### Step 5: Re-submit (Rounds 2–5)

Re-run the same review command (it reads the current diff each time, so a fresh call picks up fixes automatically):

```bash
codex review --uncommitted \
  -m o3 \
  -o "$VERDICT_FILE" \
  "Round N re-review. I fixed the issues you flagged. Please re-review the updated diff.

If the code is now solid, end with: VERDICT: APPROVED
If more changes needed, end with: VERDICT: REVISE"
```

Then go back to **Step 3**.

### Step 6: Present Final Result

Same format as Plan Review Step 7, but labeled "Code Review".

### Step 7: Cleanup

```bash
rm -f "$VERDICT_FILE"
```

---

## Rules

- Claude **actively revises** plans and **actively fixes** code between rounds — this is not just message passing
- Always use read-only sandbox for plan review (`-s read-only`)
- Max 5 rounds to prevent infinite loops
- Show the user each round's feedback and revisions so they can follow along
- If Codex CLI is not installed or fails, tell the user: `npm install -g @openai/codex`
- If a Codex suggestion contradicts the user's explicit requirements, skip it and note why
- Use fresh `codex exec` calls per round (not session resume) for reliability
